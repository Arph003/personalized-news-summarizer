Project overview

Problem, approach, and objectives

Modern users are overwhelmed by long-form online news. Skimming multiple outlets every day is time-consuming, and manual summarization is impractical. Existing tools can summarize, but they are often opaque, hard to evaluate, or tightly coupled to proprietary black-box APIs.

This project builds a lightweight, transparent summarization pipeline that can be run locally. Users paste a URL or raw article text into a simple web interface. The backend scrapes and cleans the content, generates an abstractive summary with a pre-trained transformer model, and reports automatic evaluation metrics so that the quality of the summary is visible instead of mysterious.

Project goals:

  • Provide a user-friendly web interface for summarizing arbitrary news articles via URL or raw text.
  • Use open-source models (T5-small) instead of closed commercial APIs to support reproducibility and transparency.
  • Integrate an evaluation module (ROUGE, BERTScore) directly into the UI so users can interpret summary quality at a glance.
  • Document the end-to-end system with a GitHub repository, project report, demo video, and public website.
System design

End-to-end architecture

The system consists of a static frontend (HTML / JavaScript) and a Flask backend API. The frontend is deployed via GitHub Pages and communicates with the local Flask server when the prototype is running. The backend orchestrates scraping, summarization, and evaluation.

1. Input & Frontend (Browser)

Users provide a news URL or paste raw article text into the web interface. A single /api/summarize endpoint is used for both modes.

Static HTML / CSS / JS Fetch API Form-based UI
2. Flask Backend & Scraper

The Flask server receives the request, uses requests and BeautifulSoup to download and parse the article, and normalizes the cleaned text for the model.

Flask API requests BeautifulSoup
3. Summarization & Evaluation

The cleaned article is summarized with a transformer (t5-small). The system then computes ROUGE-1/2/L and BERTScore to evaluate the generated summary.

Experiment & evaluation

Metrics: ROUGE and BERTScore

To validate the pipeline, we ran the system on the IBM article “What Are Foundation Models?”. The backend records text lengths, runtime, ROUGE scores, and BERTScore. The example below illustrates a short, highly compressed summary with moderate semantic alignment.

Metric F1 Precision Recall
ROUGE-1 0.0437 1.0000 0.0223
ROUGE-2 0.0414 0.9730 0.0212
ROUGE-Lsum 0.0437 1.0000 0.0223
BERTScore 0.4062 0.7662 0.1993
Word counts: original 1,643 · summary 41 · vocabulary coverage 97.4%

Observations:

  • The compression ratio is about 2.3%, which is intentionally aggressive to test whether the model can preserve core ideas in a very short summary.
  • High precision but low recall in ROUGE indicates that the summary uses important words correctly but omits many details from the reference text.
  • BERTScore F1 around 0.41 suggests partial semantic alignment: the model captures the main idea (“foundation models as building blocks”) but loses many nuances and examples.
  • Overall, this setup provides a starting point for evaluating different models, compression ratios, and prompt strategies in future work.
Discussion & outlook

What we learned and what comes next

Building the Personalized News Summarizer showed that it is feasible to combine open-source models, classical web scraping, and automatic evaluation into a compact prototype. Even with a relatively small model like T5-small, the system can produce readable summaries and compute basic quality metrics in a few seconds on a laptop GPU (Apple MPS).

The main limitations come from noisy HTML content, hallucinations in abstractive summarization, and the difficulty of interpreting ROUGE scores without human references. These limitations motivated the integration of BERTScore and a focus on explainable meta-information such as compression ratio and token statistics.

Concrete future directions:

  • Support batch summarization for RSS feeds or daily reading lists.
  • Experiment with larger or instruction-tuned models for higher-quality summaries while measuring runtime trade-offs.
  • Add simple preference learning, allowing users to choose summary length, style, or focus (e.g., “technical”, “business”, “beginner”).
  • Log anonymized summaries and evaluations for longitudinal analysis of model behavior across topics and time.
